LEC 01

Introduction to LangChain

Theory Code

What is LangChain?

LangChain is an open-source framework designed to build applications powered by Large Language Models. It provides composable building blocks so you can chain together prompts, models, memory, tools, and external data sources.

📌 Core Idea

Instead of calling the OpenAI API directly and gluing everything yourself, LangChain gives you components that snap together like Lego bricks.

The Problem LangChain Solves

  • Raw LLM APIs are stateless — they have no memory, no tools, no data access
  • Building real apps requires: prompt management, memory, retrieval, output parsing, tool use
  • LangChain provides all of these as composable abstractions

Installation

pip install langchain langchain-openai langchain-community python-dotenv

First API call with LangChain

from langchain_openai import ChatOpenAI
from dotenv import load_dotenv
import os

load_dotenv()

model = ChatOpenAI(
    model="gpt-4o",
    temperature=0.7
)

response = model.invoke("What is LangChain?")
print(response.content)

LEC 02

LangChain Components

Theory

The 6 Core Component Types

LangChain's architecture revolves around six composable building blocks:

Models Prompts Chains Memory Retrievers Agents

1. Models

Wrappers around LLMs and Chat Models. Two types:

  • LLMs — take a string, return a string (llm.invoke("..."))
  • Chat Models — take a list of messages, return a message (chat.invoke([HumanMessage(...)]))

2. Prompts

Template-based prompt management. PromptTemplate for LLMs, ChatPromptTemplate for Chat Models. They prevent hardcoding and allow dynamic variable injection.

3. Output Parsers

Transform raw model output into structured Python objects — strings, JSON, Pydantic models, lists, etc.

4. Chains (LCEL)

The | pipe operator connects components: prompt | model | parser. This is LCEL — LangChain Expression Language.

5. Retrievers & Vector Stores

Components that fetch relevant documents from external data sources to augment LLM context. Core of RAG architecture.

6. Agents

LLMs that can decide which tools to call based on the user's query and tool descriptions. They loop until the task is complete.

User Query → Prompt Template (formats query) → Chat Model (generates response) → Output Parser (structures output) → Final Answer
🔑 Key Principle

Every LangChain component is a Runnable. This means they all share a standard interface: .invoke(), .stream(), .batch(). This is what makes the pipe | operator work.

LEC 04

Prompts in LangChain

TheoryCode

Why Prompt Templates?

Hard-coding prompts as strings is messy and not reusable. LangChain's Prompt Templates let you define a prompt once with variables, then fill in values at runtime.

PromptTemplate (for LLMs)

from langchain_core.prompts import PromptTemplate

template = PromptTemplate(
    input_variables=["topic", "language"],
    template="Explain {topic} in {language} in simple terms."
)

# Fill in values
prompt = template.invoke({"topic": "recursion", "language": "Hindi"})

ChatPromptTemplate (for Chat Models)

from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a {role} assistant."),
    ("human",  "Explain {concept} with a real-world example.")
])

chain = prompt | model
result = chain.invoke({"role": "teaching", "concept": "polymorphism"})

MessagesPlaceholder

Used to insert a list of chat history messages (memory) into the prompt dynamically.

from langchain_core.prompts import MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}")
])

Few-Shot Prompting

Provide examples in the prompt to guide model behavior:

from langchain_core.prompts import FewShotChatMessagePromptTemplate

examples = [
    {"input": "2+2", "output": "4"},
    {"input": "capital of France", "output": "Paris"}
]
🔑 Key Insight

A ChatPromptTemplate is a Runnable. Calling .invoke(dict) returns a ChatPromptValue (list of messages), which can be piped directly into a model.

LEC 05

Structured Output in LangChain

TheoryCode

The Problem

LLMs return free-form text. Real applications need structured data — JSON, typed Python objects, specific fields. Structured Output solves this.

Method 1: .with_structured_output()

The cleanest modern way. Pass a Pydantic model or TypedDict and the LLM will constrain its output to match:

from pydantic import BaseModel, Field

class MovieReview(BaseModel):
    title: str = Field(description="Title of the movie")
    rating: int = Field(description="Rating from 1 to 10")
    summary: str = Field(description="One sentence summary")

structured_model = model.with_structured_output(MovieReview)
result = structured_model.invoke("Review Interstellar")

print(result.title)   # "Interstellar"
print(result.rating)  # 9

How it works under the hood

LangChain uses function calling / tool use (supported by OpenAI, Anthropic, Google) to force structured output. The Pydantic model is converted to a JSON schema, sent as a tool definition, and the model fills in the fields.

Method 2: TypedDict

from typing import TypedDict

class Joke(TypedDict):
    setup: str
    punchline: str

structured_model = model.with_structured_output(Joke)
result = structured_model.invoke("Tell me a programming joke")

Pydantic vs TypedDict

FeaturePydanticTypedDict
Validation✅ Full validation❌ No validation
Default values✅ Yes❌ No
Field descriptions✅ Field()❌ No
Use whenProduction appsQuick prototypes
✅ Best Practice

Always use Pydantic with Field(description=...). The descriptions are injected into the JSON schema — better descriptions = better model accuracy on filling fields.

LEC 06

Output Parsers

TheoryCode

What are Output Parsers?

Output Parsers are Runnables that sit at the end of a chain to transform raw LLM text output into usable Python objects. They're separate from Structured Output — they work by parsing text, not via function calling.

StrOutputParser

The simplest parser — extracts .content from an AIMessage and returns a plain string:

from langchain_core.output_parsers import StrOutputParser

chain = prompt | model | StrOutputParser()
result = chain.invoke({"topic": "sorting algorithms"})
print(result)  # plain string, not AIMessage

JsonOutputParser

Parses JSON from the model response. You can optionally provide a Pydantic schema for additional validation:

from langchain_core.output_parsers import JsonOutputParser

parser = JsonOutputParser(pydantic_object=MovieReview)

# The parser also generates format instructions
format_instructions = parser.get_format_instructions()

prompt = ChatPromptTemplate.from_messages([
    ("human", "Review {movie}.\n{format_instructions}")
]).partial(format_instructions=format_instructions)

chain = prompt | model | parser

PydanticOutputParser

from langchain_core.output_parsers import PydanticOutputParser

parser = PydanticOutputParser(pydantic_object=MovieReview)
chain = prompt | model | parser
result = chain.invoke({"movie": "Inception"})
print(result.rating)  # Pydantic validated int

CommaSeparatedListOutputParser

from langchain_core.output_parsers import CommaSeparatedListOutputParser

parser = CommaSeparatedListOutputParser()
chain = prompt | model | parser
result = chain.invoke({"topic": "Python frameworks"})
# returns: ["Django", "Flask", "FastAPI", "LangChain"]
📌 Output Parsers vs Structured Output

Structured Output uses native function calling — more reliable but requires model support. Output Parsers parse text — works with any model including open-source ones, but can fail if the model doesn't format correctly.

LEC 07

Chains in LangChain

TheoryCode

What is a Chain?

A chain is a sequence of components where each component's output becomes the next component's input. In modern LangChain, chains are built using LCEL and the | pipe operator.

Simple Chain

chain = prompt | model | StrOutputParser()

# Invocation
result = chain.invoke({"input": "What is machine learning?"})

Sequential Chain

The output of one chain feeds into another. Use RunnablePassthrough to carry original inputs forward:

from langchain_core.runnables import RunnablePassthrough

# Step 1: Generate a topic
chain1 = prompt1 | model | StrOutputParser()

# Step 2: Elaborate on the topic
chain2 = (
    {"topic": chain1, "original": RunnablePassthrough()}
    | prompt2
    | model
    | StrOutputParser()
)

Parallel Chains with RunnableParallel

from langchain_core.runnables import RunnableParallel

# Run two chains simultaneously
parallel = RunnableParallel(
    pros=pros_chain,
    cons=cons_chain
)
result = parallel.invoke({"topic": "Python"})
# result = {"pros": "...", "cons": "..."}
🔑 LCEL Principle

Every component in LCEL is a Runnable. The | operator calls .pipe() under the hood. A dict like {"key": runnable} is automatically treated as a RunnableParallel.

LEC 08

Runnables in LangChain — Part 1

TheoryCode

The Runnable Interface

Everything in LCEL implements the Runnable interface. This means every component exposes:

  • .invoke(input) — single input, single output (synchronous)
  • .stream(input) — yields output tokens/chunks as they arrive
  • .batch(inputs) — list of inputs → list of outputs (parallel)
  • .ainvoke(input) — async version of invoke
  • .astream(input) — async streaming

RunnableLambda

Wrap any Python function as a Runnable to use in a chain:

from langchain_core.runnables import RunnableLambda

def word_count(text: str) -> int:
    return len(text.split())

counter = RunnableLambda(word_count)

# Now usable in a chain
chain = prompt | model | StrOutputParser() | counter
result = chain.invoke({"topic": "Python"})
# result is an int — word count of the response

RunnablePassthrough

Passes the input through unchanged. Critical for preserving original values in parallel chains:

from langchain_core.runnables import RunnablePassthrough

chain = {
    "question": RunnablePassthrough(),  # original input
    "context": retriever               # retrieved docs
} | prompt | model | StrOutputParser()

RunnableParallel

Runs multiple runnables in parallel and merges their outputs into a dict. Great for branching logic:

parallel_chain = RunnableParallel({
    "english": english_chain,
    "hindi": hindi_chain,
    "summary": summary_chain
})
result = parallel_chain.invoke({"text": "Explain RAG"})
📌 Dict Shorthand

When you write {"key": runnable} as a step in a chain, LangChain auto-wraps it in RunnableParallel. It's syntactic sugar.

LEC 09

Runnables — Part 2

CodeTheory

RunnableBranch

Conditional logic in chains. Routes input to different sub-chains based on conditions:

from langchain_core.runnables import RunnableBranch

branch = RunnableBranch(
    (lambda x: x["topic"] == "math",   math_chain),
    (lambda x: x["topic"] == "code",   code_chain),
    default_chain   # fallback
)

RunnableWithFallbacks

Add error handling to any runnable — if the primary fails, fallback to another:

robust_chain = gpt4_chain.with_fallbacks([gpt35_chain, claude_chain])

RunnableRetry

resilient_chain = chain.with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True
)

itemgetter for dict access

from operator import itemgetter

chain = (
    {
        "question": itemgetter("question"),
        "context":  itemgetter("context") | format_docs
    }
    | prompt | model | StrOutputParser()
)

Batch Processing

questions = [
    {"topic": "Python"},
    {"topic": "LangChain"},
    {"topic": "RAG"}
]
# Processes all 3 in parallel
results = chain.batch(questions, config={"max_concurrency": 3})
🔑 Summary: Runnable Types

RunnableLambda — wrap a function · RunnablePassthrough — pass input unchanged · RunnableParallel — run in parallel · RunnableBranch — conditional routing · .with_fallbacks() — error resilience · .with_retry() — auto-retry

LEC 10

Document Loaders

RAGCode

What are Document Loaders?

Document Loaders ingest data from various external sources and return a list of Document objects. Each Document has:

  • page_content — the actual text
  • metadata — dict with source info (filename, page number, URL, etc.)
External Source (PDF / URL / DB / CSV / YouTube) ↓ DocumentLoader List[Document(page_content=..., metadata={...})]

Common Document Loaders

# PDF
from langchain_community.document_loaders import PyPDFLoader
loader = PyPDFLoader("document.pdf")
docs = loader.load()  # one Document per page

# Web URL
from langchain_community.document_loaders import WebBaseLoader
loader = WebBaseLoader("https://python.langchain.com/docs/...")
docs = loader.load()

# CSV
from langchain_community.document_loaders import CSVLoader
loader = CSVLoader("data.csv")
docs = loader.load()

# YouTube Transcript
from langchain_community.document_loaders import YoutubeLoader
loader = YoutubeLoader.from_youtube_url(
    "https://youtube.com/watch?v=...",
    add_video_info=True
)
docs = loader.load()

Load vs Lazy Load

  • .load() — loads all documents into memory at once
  • .lazy_load() — generator, memory-efficient for large files
  • .load_and_split(splitter) — load and immediately chunk

Directory Loader

from langchain_community.document_loaders import DirectoryLoader

loader = DirectoryLoader(
    "./docs/",
    glob="**/*.pdf",
    loader_cls=PyPDFLoader
)
docs = loader.load()  # loads all PDFs recursively
✅ Best Practice

Always inspect docs[0].metadata after loading to understand what source info is available — it's useful for citations in RAG responses.

LEC 11

Text Splitters

RAGCode

Why Split Documents?

LLMs have a context window limit. A 500-page PDF can't be sent as-is. We split it into smaller chunks that fit in the context window, then embed and store each chunk for retrieval.

⚠️ Chunking is Critical

Chunk size and overlap dramatically affect RAG quality. Too small = context loss. Too large = irrelevant content retrieved. Typical: chunk_size=1000, chunk_overlap=200.

RecursiveCharacterTextSplitter (Recommended)

Splits by a priority list of separators: ["\n\n", "\n", " ", ""]. Tries to keep paragraphs together, then lines, then words.

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,      # characters per chunk
    chunk_overlap=200,    # overlap to preserve context
    length_function=len,
    add_start_index=True  # adds position metadata
)

chunks = splitter.split_documents(docs)
print(len(chunks))                  # number of chunks
print(chunks[0].page_content)       # first chunk text
print(chunks[0].metadata)           # carries original metadata

CharacterTextSplitter

Simpler — splits on a single separator character (default "\n\n"):

from langchain_text_splitters import CharacterTextSplitter

splitter = CharacterTextSplitter(separator="\n", chunk_size=500, chunk_overlap=50)

TokenTextSplitter

Splits by tokens instead of characters. More accurate since LLMs think in tokens, not chars:

from langchain_text_splitters import TokenTextSplitter

splitter = TokenTextSplitter(chunk_size=256, chunk_overlap=32)

Chunk Size Decision Guide

LEC 12

Embeddings

RAGTheoryCode

What are Embeddings?

Embeddings convert text into dense numerical vectors in a high-dimensional space. Semantically similar texts produce vectors that are close together (measured by cosine similarity or dot product).

"What is machine learning?" → [0.23, -0.14, 0.87, ..., 0.04] (1536 dims) "Explain ML to a beginner" → [0.21, -0.16, 0.85, ..., 0.03] (similar!) "What is the weather today?" → [-0.54, 0.93, -0.21, ..., 0.77] (different)

Why Embeddings for RAG?

When a user asks a question, we embed the question and find the most similar document chunks using vector similarity search. This is semantic search — much better than keyword search.

OpenAI Embeddings

from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# Embed a query
query_vector = embeddings.embed_query("What is LangChain?")
print(len(query_vector))  # 1536

# Embed documents (batch)
doc_vectors = embeddings.embed_documents([
    "LangChain is a framework...",
    "Python is a programming language..."
])

HuggingFace Embeddings (Free)

from langchain_community.embeddings import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)

LEC 13

Vector Stores

RAGCode

What is a Vector Store?

A Vector Store (Vector Database) stores embedding vectors alongside their original text and metadata, and enables fast similarity search over millions of vectors using ANN (Approximate Nearest Neighbor) algorithms.

FAISS (In-Memory, Local)

from langchain_community.vectorstores import FAISS

# Build from documents
vectorstore = FAISS.from_documents(
    documents=chunks,
    embedding=embeddings
)

# Similarity search
results = vectorstore.similarity_search("What is RAG?", k=4)

# With scores
results = vectorstore.similarity_search_with_score("What is RAG?", k=4)

# Save and load
vectorstore.save_local("faiss_index")
vectorstore = FAISS.load_local("faiss_index", embeddings,
                                allow_dangerous_deserialization=True)

Chroma (Persistent, Local)

from langchain_chroma import Chroma

# Create with persistence
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)

# Load existing
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings
)

LEC 14

Retrievers

RAGTheoryCode

What is a Retriever?

A Retriever is a Runnable that takes a string query as input and returns a List[Document]. It's more abstract than a vector store — you don't have to specify the search method, just pass the query.

# Convert a vector store to a retriever
retriever = vectorstore.as_retriever(
    search_type="similarity",  # or "mmr", "similarity_score_threshold"
    search_kwargs={"k": 4}
)

docs = retriever.invoke("What is LangChain?")

Search Types

  • similarity — returns top k most similar documents (default)
  • mmr (Maximal Marginal Relevance) — balances relevance AND diversity; avoids returning 4 nearly-identical chunks
  • similarity_score_threshold — only return docs above a minimum similarity score

MultiQueryRetriever

Uses the LLM to auto-generate multiple variations of the query, then retrieves for each and deduplicates. Overcomes limitations of single-query retrieval:

from langchain.retrievers import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(),
    llm=model
)
# For "What is RAG?", it might also search:
# "How does Retrieval-Augmented Generation work?"
# "Explain the RAG architecture"

ContextualCompressionRetriever

Post-processes retrieved docs to extract only the relevant portion, reducing noise sent to the LLM:

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

compressor = LLMChainExtractor.from_llm(model)
retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

Other Retriever Types

WikipediaRetriever ArxivRetriever BM25Retriever (keyword) EnsembleRetriever (hybrid) ParentDocumentRetriever SelfQueryRetriever
🔑 Retriever = Runnable

Since retrievers implement the Runnable interface, they can be piped directly into chains: retriever | format_docs | prompt | model | parser

LEC 15

Building a Full RAG Pipeline

RAGProjectCode

RAG Architecture

[INDEXING PHASE - run once] Documents → Loader → Splitter → Embeddings → Vector Store [RETRIEVAL PHASE - run per query] User Query → Embed Query → Vector Similarity Search → Top K Docs [GENERATION PHASE] (Query + Top K Docs) → Prompt → LLM → Answer

Complete RAG Implementation

from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# ── 1. LOAD ──
loader = PyPDFLoader("my_doc.pdf")
docs = loader.load()

# ── 2. SPLIT ──
splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
chunks = splitter.split_documents(docs)

# ── 3. EMBED + STORE ──
embeddings = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# ── 4. PROMPT ──
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are an assistant answering questions based on the provided context.
If the answer is not in the context, say you don't know.

Context:
{context}"""),
    ("human", "{question}")
])

# ── 5. FORMAT DOCS ──
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

# ── 6. RAG CHAIN ──
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o")
    | StrOutputParser()
)

answer = rag_chain.invoke("What does the document say about RAG?")
print(answer)
✅ Pro Tip

Add source citations to your RAG: return (answer, docs) using RunnableParallel so the user knows which chunks were used to generate the response.

LEC 16

Chat History & Memory

TheoryCode

The Problem: LLMs are Stateless

Each call to an LLM is independent — it has no memory of previous turns. To build conversational apps, you must manually include chat history in each prompt.

Approach 1: Manual History

from langchain_core.messages import HumanMessage, AIMessage

history = []

def chat(user_input):
    history.append(HumanMessage(content=user_input))
    response = model.invoke(history)
    history.append(AIMessage(content=response.content))
    return response.content

Approach 2: RunnableWithMessageHistory

The LangChain way to add session-based memory to any chain:

from langchain_core.chat_history import BaseChatMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# Session store
store = {}

def get_history(session_id: str) -> BaseChatMessageHistory:
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

# Prompt with history placeholder
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}")
])

chain = prompt | model | StrOutputParser()

# Wrap with history
chain_with_history = RunnableWithMessageHistory(
    chain,
    get_history,
    input_messages_key="input",
    history_messages_key="history"
)

# Use — session_id tracks the conversation

LEC 17

Tool Calling in LangChain

AgentCode

What is Tool Calling?

Tool calling (aka function calling) allows the LLM to decide to call an external function when it needs to. The model outputs a structured request to call a tool with specific arguments — the app then executes it and returns the result.

User: "What's the weather in Delhi?" → LLM decides: I should call get_weather(city="Delhi") → App calls the actual weather API → Result returned to LLM: "28°C, sunny" → LLM responds: "It's 28°C and sunny in Delhi!"

Defining Tools

from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """Get the current weather for a given city."""
    # In reality, call a weather API
    return f"28°C and sunny in {city}"

@tool
def search_web(query: str) -> str:
    """Search the web for current information."""
    return f"Results for: {query}"

tools = [get_weather, search_web]

Binding Tools to a Model

model_with_tools = model.bind_tools(tools)

response = model_with_tools.invoke("What's the weather in Mumbai?")

# Check if model wants to call a tool
print(response.tool_calls)
# [{'name': 'get_weather', 'args': {'city': 'Mumbai'}, 'id': 'call_abc123'}]

Manual Tool Execution

from langchain_core.messages import ToolMessage

for tool_call in response.tool_calls:
    tool_name = tool_call["name"]
    tool_args = tool_call["args"]
    
    # Find and execute the tool
    tool_map = {t.name: t for t in tools}
    result = tool_map[tool_name].invoke(tool_args)
    
    # Add result back to conversation
    messages.append(ToolMessage(content=result, tool_call_id=tool_call["id"]))
📌 Tool vs Agent

Manual tool calling is a single round-trip. An Agent loops — it calls tools repeatedly, evaluates results, and keeps going until the task is done.

LEC 18

Building an End-to-End AI Agent

AgentProjectCode

What is an Agent?

An Agent is a loop where the LLM decides what to do next — call a tool, call another tool, or respond to the user. It keeps looping until it decides the task is done. The LLM is the "brain"; tools are its "hands".

User Input → LLM (with tools bound) → Tool call? YES → Execute Tool → Feed result back → LLM → Tool call? NO → Final Answer → User

ReAct Agent Pattern

ReAct = Reasoning + Acting. The model alternates between thinking (reasoning) and acting (calling tools), making the decision process transparent.

Building an Agent with LangChain

from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# Tools
tools = [get_weather, search_web, calculator_tool]

# Agent prompt (must have agent_scratchpad)
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful AI assistant with access to tools."),
    ("human", "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad")
])

# Create agent
agent = create_tool_calling_agent(model, tools, prompt)

# Executor runs the agent loop
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    verbose=True,      # see reasoning steps
    max_iterations=10  # prevent infinite loops
)

result = executor.invoke({"input": "What's the weather in Delhi and search for latest LangChain news?"})
print(result["output"])

Agent with Memory

agent_with_memory = RunnableWithMessageHistory(
    executor,
    get_history,
    input_messages_key="input",
    history_messages_key="chat_history"
)
🔑 Key Design Decisions

Write clear tool docstrings — the LLM reads them to decide which tool to use. Vague descriptions lead to wrong tool selection. Set max_iterations to prevent runaway agents. Use verbose=True during development to see the reasoning chain.

LEC 19

Project: YouTube RAG Chatbot

ProjectRAGCode

Project Overview

Build a chatbot that answers questions about any YouTube video using its transcript. Combines: YoutubeLoader → Splitter → Embeddings → FAISS → RAG Chain with Memory.

Full Implementation

from langchain_community.document_loaders import YoutubeLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory

# ── Step 1: Load YouTube transcript ──
loader = YoutubeLoader.from_youtube_url(
    "https://youtube.com/watch?v=YOUR_VIDEO_ID",
    add_video_info=True,
    language=["en", "hi"]
)
docs = loader.load()

# ── Step 2: Chunk ──
splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
chunks = splitter.split_documents(docs)

# ── Step 3: Embed & Store ──
vectorstore = FAISS.from_documents(chunks, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# ── Step 4: Prompt ──
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are an assistant that answers questions about a YouTube video.
Use the following transcript excerpts to answer questions.
If you don't know, say so — don't make things up.

Context from video:
{context}"""),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{question}")
])

# ── Step 5: Chain ──
def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

rag_chain = (
    {
        "context": retriever | format_docs,
        "question": RunnablePassthrough()
    }
    | prompt
    | ChatOpenAI(model="gpt-4o")
    | StrOutputParser()
)

# ── Step 6: Add memory ──
store = {}
def get_session(session_id):
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

chatbot = RunnableWithMessageHistory(
    rag_chain, get_session,
    input_messages_key="question",
    history_messages_key="history"
)

# ── Usage ──
config = {"configurable": {"session_id": "user_1"}}
answer = chatbot.invoke("What is this video about?", config=config)

Optional: Streamlit UI

import streamlit as st

st.title("YouTube RAG Chatbot")

LEC 21

Advanced RAG Techniques

RAGAdvanced

Why Basic RAG Fails

Basic RAG has known failure modes: poor chunk quality, single-query limitations, irrelevant retrievals, and no quality feedback loop. Advanced RAG addresses these:

1. Hybrid Search (BM25 + Vector)

Combine keyword search (BM25) with semantic search (vector) using EnsembleRetriever:

from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever

bm25 = BM25Retriever.from_documents(chunks, k=4)
vector = vectorstore.as_retriever(search_kwargs={"k": 4})

hybrid = EnsembleRetriever(
    retrievers=[bm25, vector],
    weights=[0.4, 0.6]  # 40% keyword, 60% semantic
)

2. Reranking (Cross-Encoder)

After retrieval, rerank results with a more accurate cross-encoder model before sending to LLM:

pip install flashrank

from langchain_community.document_compressors import FlashrankRerank
from langchain.retrievers import ContextualCompressionRetriever

reranker = FlashrankRerank(top_n=3)
retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=hybrid
)

3. Parent Document Retriever

Embed small chunks (for precise matching) but return the full parent document (for context):

from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore

store = InMemoryStore()
retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=store,
    child_splitter=RecursiveCharacterTextSplitter(chunk_size=400),
    parent_splitter=RecursiveCharacterTextSplitter(chunk_size=2000)
)

4. HyDE (Hypothetical Document Embeddings)

Generate a hypothetical answer to the query, embed that, and use it for retrieval. Often more semantically aligned than embedding the raw question:

# Generate hypothetical answer
hyde_prompt = ChatPromptTemplate.from_template(
    "Write a short paragraph that answers: {question}"
)
hyde_chain = hyde_prompt | model | StrOutputParser()

hypothetical_doc = hyde_chain.invoke({"question": user_query})
docs = retriever.invoke(hypothetical_doc)  # embed hypothetical, not query

Advanced RAG Architecture Map

Query → MultiQuery (generate N variations) → HyDE (generate hypothetical answer) → Hybrid Search (BM25 + Vector) → Reranker (Cross-Encoder) → Context Compression → RAG Chain → Answer
